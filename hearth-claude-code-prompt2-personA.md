## ROLE & MISSION

You are building the Gradient AI graph for **Hearth**, a benefits screener. The graph must: extract a structured `HouseholdProfile` from messy, possibly non-English free text; route to a domain agent; call Person B's deterministic `/screen` function; and explain the result in warm, plain, **cited** language that never overstates certainty.

The model does language only. It must never assert an eligibility outcome, a dollar figure, or a guarantee — those come from `/screen`.

Success = a judge types a plain-language situation and gets a real, cited screen back, and a live guardrail visibly rewrites a "guaranteed $X" hallucination into honest estimate language.

1. **Feature branch** `feat/p2-gradient-graph`; commit at each VERIFY GATE; never touch main directly.
2. **State-aware first.** Before creating any Gradient resource, list what already exists (agents, KBs, guardrails) and reconcile — do not create duplicates. Report what you found.
3. **Stack.** TypeScript. Use the official **DigitalOcean TypeScript SDK (DoTs)** for Gradient resources where it supports the operation; fall back to the Gradient REST API with **native fetch + AbortController (never axios)**. **Any step that must be done in the DO Control Panel console → flag it explicitly with the exact console path** so it's reproducible.

6. **Adversarial verification.** Gates must include the guarantee-rewrite and a missing-input case, not just one happy path.
7. **Surgical + reproducible.** Every resource you create is scripted or documented so it can be recreated; no click-once-and-forget config.
8. **Stop at every VERIFY GATE**; report pass/fail per check; then stop and report at the end. Do not build the frontend, the filer, or Person B's engine.

## PHASE A1 — the graph

**Task A1.1 — Intake agent.** Create a Gradient agent whose sole job is: free text (any language) → strict JSON matching `HouseholdProfile`. System prompt rules: extract only what's stated; **missing fields → null, never inferred or guessed**; normalize currency/frequency (e.g., "$650/week" → monthly gross); detect `preferredLanguage`. Output must be parseable JSON, nothing else.

**Task A1.2 — Knowledge Base (food/CalFresh).** Create a KB; add the official CalFresh sources as crawl seeds: `sfhsa.org` (CalFresh pages) and `cdss.ca.gov`. **Start indexing immediately** so it finishes while you build the rest. Choose an embedding model per DO guidance; keep the KB in the same region as the agents to reduce latency.

**Task A1.3 — Food domain agent.** Create the Food agent. Instructions: (a) receive the `HouseholdProfile`, (b) call the **function route** to `/screen`, (c) explain the returned `ScreeningResult` in warm, plain language, (d) **always** include the `disclaimer` and surface the real `citations`, (e) attach the food KB for grounding, (f) never state a benefit as guaranteed or certain. Route more domain agents in later (Person B is adding health/utilities/tax programs in Prompt 3) — structure so they slot in without rework.

**Task A1.4 — Function route.** Register `/screen` (`SCREEN_URL`, or the mock for now) as a callable function on the Food agent; map `HouseholdProfile` → request body; parse `ScreeningResult[]` back.

**Task A1.5 — Router agent.** Create a router that classifies intent and routes to the Food agent now, with clean extensibility to health/utilities/tax agents. The entry point the frontend calls is the router.

> **VERIFY GATE A1 (adversarial):**
> - "single mom in SF, about $2,800 a month, one kid, renting" → intake returns a correct profile (household 2, gross 2800, renter, child) with unstated fields null → router → Food → `/screen` (mock ok) → a plain-language, **cited** answer with the disclaimer. Capture the full trace.
> - Non-English: the same in Spanish → correct profile + `preferredLanguage: 'es'`.
> - Missing income ("I have two kids and rent is high") → agent returns `need_more_info` / asks, does **not** fabricate a number.
> Report the router entry ID, KB indexing status, and any console-only steps. Commit.
